# Generative AI Portfolio — Project Brief

## What this is

This is **Erik Ferrari (Erikwski)'s personal landing page**, built as a **Generative UI application** powered by [Hashbrown](https://hashbrown.dev) and Angular SSR.

The core idea: **the UI itself is the AI output**. There is no fixed layout. Claude composes Angular components dynamically in response to the visitor's prompts, personalized to who they are.

---

## User Flow

### 1. Onboarding screen

The visitor lands on a minimal intake form asking who they are:

- A **peer developer**
- A **recruiter**
- A **potential client**

Other information: 
The Question: "How much time do you have for me today?"
Options: ⚡ Elevator Pitch (30s) | ☕ Coffee Break (5m) | 🔍 Deep Dive (15m+)

The Question: "How should we talk?"
Options: 💼 Formal & Precise | 🎨 Creative & Experimental | ⚡ Fast & Technical

This data is submitted and stored in a **global NgRx Signal Store**. It persists for the full session and drives personalization across all generated components.

### 2. Generative canvas screen

After submission the visitor reaches the main experience:

- A **text input** (centered, fullscreen) with a **send button**
- **3 suggestion chips** with default starter questions (e.g. "What can you build?", "Show me your stack", "Are you available for hire?")
- No components are rendered yet — the canvas is empty

### 3. Component generation

When the visitor sends a prompt:

- Claude (via Hashbrown) generates one or more UI components as a response
- Components appear inside a **draggable grid** (the canvas)
- The text input slides to the bottom of the screen as components fill the canvas
- Each generated component is aware of the visitor's profile (from the signal store) and adapts its content accordingly — e.g. the About component shows different copy for a recruiter vs. a peer dev

### 4. Persistent controls

At all times the visitor can:

- **Edit their profile data** (reopens the onboarding form, updates the signal store)
- **Reset the experience** (clears the canvas and the store, returns to step 1)

---

## Personalization logic

The visitor's profile (role + any extra context they provide) is stored globally and injected into every Hashbrown prompt as system-level context. Components are not hardcoded — Claude decides what to render and how to frame the content based on who is asking.

## Folder structure

```
src/app/
├── shared/          # Shared components, utilities, domain models used across the app
├── pages/           # Route-level components (one per screen)
│   ├── onboarding/  # Step 1: visitor profile form
│   ├── canvas/      # Step 2: empty canvas with centered text input + chips
│   └── experience/  # Step 3: text input at bottom + draggable widget grid
└── widgets/         # Components registered with Hashbrown and generated by Claude
```

Each folder (shared, page, widget) can grow its own internal layers **only when needed**:

```
<feature>/
├── ui/          # Dumb/presentational components — no side effects, inputs/outputs only
├── feature/     # Smart components — orchestrate state and services
├── utils/       # Pure functions, helpers, pipes
├── domain/      # TypeScript interfaces, types, enums
└── data-access/ # Store access and API calls (signal store, HTTP)
```

Don't create these sub-layers preemptively — only introduce a layer when a second file of that type is needed.

## Tech stack

- **Angular 21** with SSR (Express)
- **Hashbrown** — Generative UI framework (Angular adapter)
- **@hashbrownai/anthropic** — Claude API integration via `POST /api/chat`
- **@ngrx/signals** — global signal store for visitor profile
- **Angular Material 21 (M3)** — component library and design system
- **Vercel** — deployment target

## Design system

**ALWAYS use Angular Material components** (`@angular/material/*`) for all UI elements — buttons, inputs, chips, cards, dialogs, toggles, etc. Never build custom interactive components from scratch when a Material equivalent exists.

Dark mode, ultra-minimalist. Theme defined in `src/material-theme.scss` via `mat.theme()` with dark `theme-type` and `mat.$red-palette` as primary.

Custom design tokens (CSS variables) are defined on `html` in `material-theme.scss` alongside the M3 overrides:
- `--color-accent` `#FF3B30` — Electric Red, UI accent
- `--color-brand` `#FFBF00` — Brand Gold, from logo gradient
- `--color-bg` `#0A0A0A`, `--color-surface` `#141414`, `--color-surface-elevated` `#1E1E1E`
- `--color-muted`, `--color-subtle`, `--color-border`, `--color-border-accent`
- `--font-sans` Inter, `--font-mono` JetBrains Mono
- `--radius-sm/md/panel`, `--shadow-panel`, `--shadow-accent`, `--transition-fast/normal`

UI language: floating "Aura:" window cards, Electric Red accent, Brand Gold from logo, Inter font.
Logo assets: `public/logo/dark/` and `public/logo/white/`, three layouts each (`logo.svg`, `logo-horizontal.svg`, `logo-mobile.svg`).
Logo component: `src/app/shared/ui/logo/logo.component.ts`.
